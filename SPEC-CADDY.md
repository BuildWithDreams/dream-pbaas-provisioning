# SPEC: Caddy Reverse Proxy with Lets Encrypt

## 1. Context

The BWD stack has two HTTP services that need to be exposed publicly with TLS:

- **RPC server** — Verus RPC, bound to `10.201.0.12:37486` on `net-vrsc-blue`
- **QR creator** — Node.js web app, bound to `10.201.0.13:3000` on `net-vrsc-blue`

Both currently bind to `127.0.0.1` (localhost only) and are not accessible externally. They need to be reachable via `https://rpc.vrsc.buildwithdreams.com` and `https://qrcodes.buildwithdreams.com` with automatic Lets Encrypt certificates.

Caddy is chosen over nginx because:
- Automatic HTTPS for new subdomains (no manual certbot)
- Built-in WebSocket support (future-proof for other services)
- Simpler configuration (single `Caddyfile`)
- Auto-renewal of certificates

## 2. Design

### 2.1 Container Identity

| Field | Value |
|-------|-------|
| Compose project | `mains_blue_caddy` |
| Container name | `mains_blue_caddy-caddy-1` |
| Docker network | `net-vrsc-blue` (external, existing) |
| Bind IP | `10.201.0.10` |
| Published port | `443` (HTTPS), `80` (HTTP → redirect) |

### 2.2 Routing Table

| Public URL | Upstream | Notes |
|-----------|----------|-------|
| `https://rpc.vrsc.buildwithdreams.com` | `http://10.201.0.12:37486` | VRSC RPC server |
| `https://qrcodes.buildwithdreams.com` | `http://10.201.0.13:3000` | QR creator web app |

### 2.3 Sensitive Variables

These values are **per-deployment** and must **never be committed**:

| Variable | Description | Stored in |
|----------|-------------|-----------|
| `caddy_letsencrypt_email` | Email for Lets Encrypt expiry notices | `group_vars/production-local.yml` |
| `caddy_domain` | Root domain for subdomains | `group_vars/production-local.yml` |
| `target_host` | Server IP | `group_vars/production-local.yml` |

### 2.4 Caddyfile Structure

```
{
  email {$CADDY_LETSENCRYPT_EMAIL}
  on_demand_tls {
    issue ecdsa
  }
}

rpc.{$CADDY_DOMAIN} {
  reverse_proxy 10.201.0.12:37486
  log {
    output file /data/access.log
  }
}

qrcodes.{$CADDY_DOMAIN} {
  reverse_proxy 10.201.0.13:3000
  log {
    output file /data/access.log
  }
}
```

The `on_demand_tls` block enables automatic certificate issuance for any subdomain pointed at this Caddy instance — future services can be added without rebuilding the container, provided DNS is pointed to the server first.

## 3. Playbook Design

### Playbook: `28-caddy-deploy.yml`

**Purpose:** Create Caddy container on `net-vrsc-blue` with `docker compose`.

**Run:** `ansible-playbook -i inventory.ini playbooks/28-caddy-deploy.yml`

**Parameters (all via `-e`):**

| Parameter | Default | Description |
|-----------|---------|-------------|
| `caddy_letsencrypt_email` | *(required)* | Email for Lets Encrypt |
| `caddy_domain` | *(required)* | Root domain e.g. `buildwithdreams.com` |

**What it does:**

1. Read `caddy_letsencrypt_email` and `caddy_domain` from playbook args (user provides at runtime, or reads from `group_vars/production-local.yml` if available)
2. Write `Caddyfile` to `caddy_path/` using template with domain variables
3. Write `docker-compose.yml` for Caddy (project `mains_blue_caddy`, network `net-vrsc-blue`, IP `10.201.0.10`, ports `80` and `443`)
4. Stop any existing Caddy container (idempotent)
5. Start Caddy via `docker compose`
6. Wait for container to be healthy
7. Display summary with subdomain URLs

**Idempotency:** Safe to re-run. Always recreates the container to pick up latest `Caddyfile`.

### Playbook: `29-caddy-teardown.yml`

**Purpose:** Remove the Caddy container and its compose project.

**Run:** `ansible-playbook -i inventory.ini playbooks/29-caddy-teardown.yml`

## 5. DNS Prerequisite

Before running the playbook, DNS must be configured:

| Record | Type | Value |
|--------|------|-------|
| `rpc.vrsc` | A | `<server IP>` |
| `qrcodes` | A | `<server IP>` |

Caddy will automatically provision TLS certificates on first request once DNS is live.

---

## 6. Out of Scope

- WebSocket routing (not needed for current services, but Caddy supports it via `websocket` directive in the `reverse_proxy` block)
- Adding new subdomains post-deployment — handled by pointing DNS at the server and making a request; Caddy's `on_demand_tls` handles issuance automatically
- Caddy metrics/monitoring
- Authentication on proxied endpoints (services handle their own auth)

## 7. Open Questions

### Q1: Service Parameter Registry (deferred — see Section 9)

Should service bind IPs, ports, and compose project names be tracked in a central file in the provisioning repo, so Caddy and future infrastructure can discover services without hardcoding values across playbooks?

Current state: each service's playbook defines its own `rpc_bind_ip`, `qr_bind_ip`, etc. as hardcoded vars. A `service-registry.yml` in `group_vars/` could enumerate all deployed services and their parameters, making Caddy's routing table data-driven rather than hardcoded.

### Adding a new service to the proxy:

1. Assign an IP in `net-vrsc-blue` (next available .14, .15, etc.)
2. Ensure the service's compose file binds to `127.0.0.1:<port>` (host) or exposes the port on the Docker network
3. Add the subdomain and upstream to the `Caddyfile` template in `28-caddy-deploy.yml`
4. Point DNS to the server — Caddy will issue the cert on first request
